# REST API

Базовый префикс: **`/api`**. Роутер контактов: **`/contacts`**. Итоговые пути: `/api/contacts`, `/api/contacts/{id}`.

Формат тела запросов и ответов — **JSON**. Кодировка UTF-8.

## Эндпоинты

### `GET /api/contacts`

Возвращает массив всех контактов.

- **Успех:** `200 OK`
- **Тело:** массив объектов `Contact` (см. ниже).
- **Порядок:** новые записи первыми (`id` по убыванию).

### `GET /api/contacts/{contact_id}`

Один контакт по числовому `id`.

- **Успех:** `200 OK`, тело — `Contact`.
- **Ошибка:** `404 Not Found` — `{"detail": "Contact not found"}`.

### `POST /api/contacts`

Создание контакта.

- **Тело:** объект `ContactCreate`.
- **Успех:** `201 Created`, тело — `Contact` с присвоенным `id` и `created_at`.

### `PUT /api/contacts/{contact_id}`

Частичное обновление: передаются только изменяемые поля (`ContactUpdate`).

- **Успех:** `200 OK`, тело — обновлённый `Contact`.
- **Ошибка:** `404 Not Found`, если записи с таким `id` нет.

В слое сервиса поле `email` меняется только если в разобранной модели `email is not None`. Поэтому передать в JSON значение `null` для сброса email **нельзя** — оно трактуется как «не обновлять» (как и отсутствие ключа). Пустая строка для `phone` в `ContactUpdate` допустима и очищает телефон.

### `DELETE /api/contacts/{contact_id}`

Удаление контакта.

- **Успех:** `204 No Content`, пустое тело.
- **Ошибка:** `404 Not Found`.

## Схемы данных

### `Contact` (ответ)

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | integer | Идентификатор |
| `full_name` | string | Имя |
| `phone` | string | Телефон |
| `email` | string \| null | Email |
| `notes` | string | Заметки |
| `created_at` | string | ISO-формат даты/времени (строка из ORM через `isoformat()`) |

### `ContactCreate` (тело `POST`)

| Поле | Обязательность | Ограничения |
|------|------------------|-------------|
| `full_name` | да | 1–200 символов |
| `phone` | нет, по умолчанию `""` | до 50 символов в схеме; если не пусто — после удаления нецифровых символов должно остаться **не меньше 10 цифр** |
| `email` | нет | Корректный email (`EmailStr`) или отсутствие поля / `null` |
| `notes` | нет, по умолчанию `""` | до 2000 символов |

Пустой телефон (`""`) валиден и не проверяется на количество цифр.

### `ContactUpdate` (тело `PUT`)

Все поля опциональны. Переданные поля обновляются; `null` для `email` и непустые строки для остального обрабатываются согласно валидаторам.

- Для `phone`: `null` — «не менять»; `""` — очистить; непустая строка — те же правила, что у create (≥10 цифр).

## Ошибка валидации (422)

При ошибке Pydantic сервер возвращает структуру:

```json
{
  "error": {
    "code": "validation_error",
    "message": "Request validation failed",
    "details": [
      {
        "field": "email",
        "message": "value is not a valid email address: ...",
        "code": "value_error"
      }
    ]
  }
}
```

Поле `field` строится из `loc` без префикса `body` (см. обработчик в `app/main.py`).

## Прочие ошибки

- **`404`** для отсутствующего контакта — стандартный `detail` FastAPI (строка или объект).
- **`500`** — JSON с `error.code = "internal_error"` и общим сообщением (детали исключения клиенту не отдаются).

## CORS

В `main.py` включён `CORSMiddleware` с `allow_origins=["*"]` — удобно для локальной разработки и произвольного origin. Для публичного продакшена обычно сужают список доменов.
